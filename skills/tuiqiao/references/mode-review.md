# 模式 E：计划评审与复评

## 触发

`tuiqiao-review <需求ID> [评审重点]`

目标：以挑剔视角尽可能找问题，并提供可落地改进建议。

---

## 评审前检查

### 必读文件

1. `requirement.md`
2. `design.md`

### 复评时额外必读

1. `plan_review.md` 既有问题清单
2. `plan_review.md` 中的修订回应记录
3. `handoff.md`

### 验收失败后复评时再额外必读

4. `acceptance_feedback.md`

---

## 必查清单

| 检查项 | 说明 |
|--------|------|
| Task可执行性 | 输入/输出/验收条件是否客观可执行 |
| 链路闭环 | 依赖和顺序是否合理 |
| 完整性 | 是否缺准备、验证、回滚、监控、交接 |
| 风险控制 | 高风险步骤是否有预案 |
| 优化空间 | 是否存在更低成本或更低风险方案 |
| 局部上下文 | 单Task是否可独立委派 |
| 会话独立性 | 新会话仅读文档能否执行 |
| 验收整改覆盖 | 是否逐条覆盖验收不通过意见 |

---

## 输出要求

1. 优先暴露问题，不做空泛表扬。
2. 每条问题必须带：定位、影响、修复建议。
3. 复评时必须给"问题闭环对照"。
4. 若有验收失败历史，必须给"验收整改覆盖对照"。
5. 使用 `template-review.md` 格式输出。

---

## 落库规则

### 首次评审

1. 若 `plan_review.md` 为空或不存在，创建文档（使用 `template-review.md` 首次评审格式）。
2. 评审结果追加写入 `plan_review.md`。
3. 更新 `meta.json` 状态。

### 复评

1. 评审结果追加写入 `plan_review.md`。
2. 更新 `meta.json` 状态。

### 状态映射

| 结论 | 状态 |
|------|------|
| 可执行 | 方案已确认 |
| 有条件可执行 | 方案待优化 |
| 不可执行 | 方案待修订 |
