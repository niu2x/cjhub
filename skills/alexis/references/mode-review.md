# 模式 E：计划评审与复评

## 触发

`alexis-review <需求ID> [评审重点]`

目标：以挑剔视角尽可能找问题，并提供可落地改进建议。

## 评审前检查

必读：

1. `requirement.md`
2. `design.md`

若信息不足，先提最多 3 个关键问题再继续。

## 复评场景（重新 review）

若是修订后复评，额外必读：

1. `plan_review.md` 既有问题清单
2. `plan_review.md` 中的修订回应记录
3. `handoff.md`

若最近验收不通过，再额外读取：

4. `acceptance_feedback.md`

## 必查清单

1. Task 可执行性：输入/输出/验收条件是否客观可执行
2. 链路闭环：依赖和顺序是否合理
3. 完整性：是否缺准备、验证、回滚、监控、交接
4. 风险控制：高风险步骤是否有预案
5. 优化空间：是否存在更低成本或更低风险方案
6. 局部上下文：单 Task 是否可独立委派
7. 会话独立性：新会话仅读文档能否执行
8. 验收整改覆盖：是否逐条覆盖验收不通过意见

## 输出要求

- 优先暴露问题，不做空泛表扬。
- 每条问题必须带：定位、影响、修复建议。
- 复评时必须给"问题闭环对照"（已解决/部分解决/未解决/新增问题）。
- 若有验收失败历史，必须给"验收整改覆盖对照"。
- **必须使用 `templates.md` 中的"评审报告模板"格式输出**。

## 落库规则

1. **首次评审**：
   - 若 `plan_review.md` 为空或不存在，先创建文档头部（参考 `templates.md` 中的"plan_review.md 文档结构"）。
   - 评审结果追加写入 `plan_review.md` 的"评审记录"区域。
   - 更新版本历史表。

2. **复评**：
   - 评审结果追加写入 `plan_review.md` 的"评审记录"区域。
   - 更新版本历史表。
   - 更新文档头部的"当前状态"和"最后更新"。

3. 更新 `meta.json` 的 `updated_at`。
4. 评审模式不自动改写 `design.md`。
5. 结论为不可执行时，建议状态 `方案待修订`；有条件可执行时建议 `方案待优化`。
